# Mandelbrot Fractal Simulation

Інтерактивна візуалізація множини Мандельброта на C++/SFML, де важка математика винесена на GPU через fragment shader.

Проект відкриває вікно 1600x900, малює один прямокутник на весь екран, а кожен піксель цього прямокутника рахується прямо у `assets/mandelbrot.frag`. C++-код у `src/main.cpp` не перебирає пікселі сам. Він керує вікном, камерою, масштабом, кількістю ітерацій, FPS-текстом і передає в shader поточні межі комплексної площини.

## Що тут відбувається

Множина Мандельброта будується дуже простою формулою:

```text
z = z^2 + c
```

Для кожної точки `c` на комплексній площині ми починаємо з `z = 0` і дивимося, чи "втече" значення за межі радіуса. Якщо втекло швидко - точка не належить множині. Якщо не втекло за `maxIt` ітерацій - вважаємо її внутрішньою і фарбуємо в чорний.

Проблема не у формулі. Проблема у кількості повторень.

Вікно має `1600 * 900 = 1 440 000` пікселів. На старті стоїть `80` ітерацій. У найгіршому випадку це до:

```text
1 440 000 * 80 = 115 200 000 ітерацій на один кадр
```

Для 60 FPS це вже мільярди маленьких операцій на секунду. Саме тому CPU-варіант швидко впирається в стелю: навіть якщо код написаний нормально, він все одно намагається кожен кадр перерахувати величезну кількість незалежних пікселів.

GPU для цього підходить набагато природніше: fragment shader запускається для кожного пікселя, і відеокарта рахує багато таких пікселів паралельно.

## Чому я відмовився від CPU на користь GPU

Спочатку можна було б рахувати фрактал на CPU: пройтись циклом по всіх `x/y`, для кожного пікселя порахувати ітерації, скласти колір у масив, завантажити цей масив у texture і показати на екрані. Це працює, але для інтерактивного zoom/pan швидко стає неприємно.

У цьому проекті пікселі не залежать один від одного. Результат для лівого верхнього пікселя ніяк не впливає на правий нижній. Така задача майже ідеально лягає на GPU.

| Питання | CPU-підхід | GPU-підхід у цьому проекті |
| --- | --- | --- |
| Хто рахує пікселі | Центральний процесор | Fragment shader на відеокарті |
| Паралельність | Обмежена кількістю ядер CPU | Масова паралельність по пікселях |
| Передача даних | Треба оновлювати texture/pixel buffer | C++ передає тільки кілька uniform-змінних |
| Реакція на рух камери | Кожен рух змушує CPU перераховувати весь кадр | GPU просто перемальовує кадр з новими межами |
| Код рендера | Більше ручної роботи з масивами кольорів | Один fullscreen rectangle + shader |
| Слабке місце | CPU час і копіювання пікселів у GPU | Обчислення у fragment shader |

Ще одна причина - простота. CPU-рендеринг швидко тягне за собою потоки, tile-based rendering, SIMD, кеші, double buffering для текстури, синхронізацію. Тут цього немає. C++ лише каже shader'у: "ось поточна область комплексної площини, ось розмір вікна, ось максимальна кількість ітерацій". Далі кожен піксель рахується сам.

| Навантаження | Значення у проекті |
| --- | ---: |
| Розмір вікна | `1600x900` |
| Кількість пікселів | `1 440 000` |
| Початкова кількість ітерацій | `80` |
| Верхня межа роботи на кадр | до `115 200 000` ітерацій |
| Цільовий ліміт кадрів | `60 FPS` |

Ці цифри не є бенчмарком. Це просто масштаб задачі. Реальна кількість ітерацій менша, бо багато точок втікають раніше, а частина внутрішніх областей відсікається окремими перевірками в shader'і.

## Структура проекту

```text
.
├── CMakeLists.txt
├── assets
│   ├── font.ttf
│   └── mandelbrot.frag
└── src
    ├── main.cpp
    └── main.guide.cpp
```

| Файл | Для чого він |
| --- | --- |
| `src/main.cpp` | Основний C++ код: вікно, input, камера, uniforms, FPS overlay |
| `src/main.guide.cpp` | Та сама логіка з коментарями для пояснення коду |
| `assets/mandelbrot.frag` | GLSL fragment shader, який реально рахує фрактал |
| `assets/font.ttf` | Шрифт для тексту з FPS, zoom та кількістю ітерацій |
| `CMakeLists.txt` | Збірка через CMake, підключення SFML, копіювання assets у build |

## Керування

| Клавіша | Дія |
| --- | --- |
| `W` / `S` | Рух вгору / вниз по комплексній площині |
| `A` / `D` | Рух вліво / вправо |
| `Up` | Наближення |
| `Down` | Віддалення |
| `[` | Зменшити кількість ітерацій |
| `]` | Збільшити кількість ітерацій |
| `R` | Скинути область перегляду |
| `Esc` | Закрити програму |

Примітка: у коді `Up` зменшує розмір області комплексної площини, тобто наближує камеру, а `Down` збільшує область, тобто віддаляє.

## Як працює `main.cpp`

`main.cpp` не малює фрактал вручну. Його задача - підготувати контекст для shader'а.

Основні частини:

| Частина | Що робить |
| --- | --- |
| `WindowSize` | Фіксує розмір вікна `1600x900` |
| `FractalView` | Зберігає межі комплексної площини: `Rm`, `RM`, `Im`, `IM` |
| `handleEvents` | Обробляє закриття вікна, `Esc`, `R` |
| `applyRealtimeInput` | Плавний рух, zoom і зміна ітерацій |
| `applyShaderUniforms` | Передає в shader розмір екрана, межі області та `maxIt` |
| `makeInfoString` / `updateFps` | Формує текст з FPS, zoom та кількістю ітерацій |

### Оптимізації в C++ частині

| Оптимізація | Навіщо вона |
| --- | --- |
| Один `sf::RectangleShape` на весь екран | Не створюємо мільйон об'єктів. Малюємо одну площину, все інше робить shader |
| Немає CPU pixel buffer | CPU не заповнює масив кольорів і не ганяє його в texture кожен кадр |
| Камера зберігається у `double` | Рух і zoom на стороні CPU точніші, ніж якби вся камера одразу жила у `float` |
| Uniforms замість перебудови shader'а | При русі змінюються тільки числа `Rm/RM/Im/IM/maxIt`, shader не компілюється заново |
| `deltaSec` для руху | Швидкість руху не залежить напряму від FPS |
| Обмеження `deltaSec` до `0.05f` | Якщо кадр підвис, камера не стрибає занадто далеко |
| Pan залежить від `view.width()` / `view.height()` | На великому zoom рух лишається природним, а не занадто грубим |
| Zoom навколо центра | Картинка не "з'їжджає" при масштабуванні |
| `setKeyRepeatEnabled(false)` | Одноразові клавіші типу `R` не спамляться повторними key events |
| Зміна `maxIt` раз на `0.05s` | При затиснутій `[` або `]` значення змінюється контрольовано |
| FPS-текст оновлюється раз на пів секунди | Не треба форматувати рядок кожен кадр |
| `setFramerateLimit(60)` | Програма не намагається гріти GPU сотнями непотрібних FPS |
| Assets копіюються після build | `mandelbrot.frag` і `font.ttf` опиняються поруч з executable |

Окремо варто згадати `FractalView`. Це маленька, але важлива структура. Вона не просто зберігає координати, а ще знає свою ширину, висоту, центр, zoom і вміє робити `reset`, `pan`, `scaleAroundCenter`. Через це логіка камери не розмазана по всьому `main`.

## Як працює `mandelbrot.frag`

Shader запускається для кожного пікселя. На вході він має:

| Uniform | Значення |
| --- | --- |
| `resolution` | Розмір вікна |
| `Rm`, `RM` | Ліва і права межа дійсної осі |
| `Im`, `IM` | Нижня і верхня межа уявної осі |
| `maxIt` | Максимальна кількість ітерацій |

### Перехід від пікселя до комплексного числа

```glsl
vec2 uv = pixel / resolution;
return vec2(
    mix(Rm, RM, uv.x),
    mix(IM, Im, uv.y)
);
```

`gl_FragCoord.xy` дає координату пікселя на екрані. Shader переводить її у `uv` від `0` до `1`, а потім через `mix` знаходить відповідну точку на комплексній площині.

По `y` використано `mix(IM, Im, uv.y)`, бо екранні координати ростуть вниз, а уявна вісь у математиці звично росте вгору.

### Оптимізації в shader'і

| Оптимізація | Що дає |
| --- | --- |
| Перевірка головної кардіоїди | Багато точок всередині множини одразу фарбуються в чорний без циклу |
| Перевірка period-2 bulb | Ще одна велика внутрішня область відсікається до ітерацій |
| Порівняння з квадратом радіуса | Немає `sqrt`, бо достатньо перевіряти `zx2 + zy2 <= 4.0` |
| Окремі `zx`, `zy`, `zx2`, `zy2` | Менше зайвих множень, квадрати повторно використовуються |
| Ранній `return` для внутрішніх точок | Shader не виконує зайву роботу після очевидного результату |
| Ранній `return`, якщо `it >= maxIt` | Точки, які не втекли, одразу стають чорними |
| Smooth coloring | Колір змінюється плавніше, без грубих смуг між ітераціями |
| Проста палітра | Колір рахується арифметикою, без texture lookup |

Найважливіша оптимізація тут - `isDefinitelyInside`.

```glsl
bool insideMainCardiod = r * (r + x) <= 0.25 * c.y * c.y;
bool insidePeriodTwoBulb = (c.x + 1.0) * (c.x + 1.0) + c.y * c.y <= 0.0625;
```

У множини Мандельброта є області, про які можна математично сказати: точка точно всередині. Якщо точка потрапила туди, немає сенсу крутити цикл до `maxIt`. Для центру картинки це дуже помітно, бо чорна внутрішня частина займає багато місця.

### Основний цикл

```glsl
while (zx2 + zy2 <= EscapeRadius && it < maxIt) {
    zy = 2.0 * zx * zy + c.y;
    zx = zx2 - zy2 + c.x;

    zx2 = zx * zx;
    zy2 = zy * zy;

    ++it;
}
```

Це та сама формула `z = z^2 + c`, тільки розкладена на дійсну та уявну частини. Так shader не створює ніяких складних об'єктів, а працює з простими `float`.

## Що можна покращити далі

У поточній версії є нормальний баланс між простотою і швидкістю, але є чесні обмеження:

| Обмеження | Чому так | Як можна покращити |
| --- | --- | --- |
| Дуже глибокий zoom почне втрачати точність | У shader передаються `float`, а не `double` | Perturbation rendering, double-float math або інша схема координат |
| Немає anti-aliasing | Кожен піксель рахується один раз | Supersampling або adaptive sampling |
| Розмір вікна фіксований | `WindowSize` заданий константою | Додати resize і оновлення `resolution` |
| Палітра проста | Колір рахується короткою формулою | Додати кілька палітр або gradient texture |
| Немає zoom до курсора | Zoom іде навколо центру | Враховувати позицію миші при масштабуванні |

## Встановлення і запуск

Проект використовує:

- C++17 compiler
- CMake `3.16+`
- SFML `3.x` з модулями `Graphics`, `Window`, `System`
- Відеодрайвер з підтримкою OpenGL shaders

Офіційні сторінки SFML, які можуть знадобитися:

- [SFML 3.0 tutorials](https://www.sfml-dev.org/tutorials/3.0/)
- [SFML with CMake](https://www.sfml-dev.org/tutorials/3.0/getting-started/cmake/)
- [SFML shaders](https://www.sfml-dev.org/tutorials/3.0/graphics/shader/)
- [SFML downloads](https://www.sfml-dev.org/download/sfml/3.0.2/)

### macOS

Встановити інструменти:

```bash
xcode-select --install
brew install cmake sfml
```

Якщо Homebrew пропонує versioned package для SFML 3, можна поставити саме його:

```bash
brew install sfml@3.0.2
```

Збірка і запуск:

```bash
mkdir -p build && cd build
cmake ..
make
./main
```

Або через універсальний CMake build command:

```bash
cmake -S . -B build
cmake --build build
cd build
./main
```

Запускати краще саме з папки `build`, бо shader і font копіюються туди під час збірки.

### Linux

Для Ubuntu/Debian спочатку поставити компілятор, CMake і SFML:

```bash
sudo apt update
sudo apt install build-essential cmake libsfml-dev
```

Важливо: проект написаний під SFML 3. Якщо твій дистрибутив ставить старий `libsfml-dev` з SFML 2.x, CMake чесно впаде на `find_package(SFML 3 ...)`. У такому випадку треба поставити SFML 3 з пакетів дистрибутива, через vcpkg або зібрати SFML з вихідного коду.

Збірка і запуск:

```bash
mkdir -p build && cd build
cmake ..
make
./main
```

Або:

```bash
cmake -S . -B build
cmake --build build
cd build
./main
```

### Windows

Найпростіший варіант - Visual Studio 2022 з workload `Desktop development with C++` і встановлений CMake.

Варіант через vcpkg:

```powershell
git clone https://github.com/microsoft/vcpkg C:\vcpkg
C:\vcpkg\bootstrap-vcpkg.bat
C:\vcpkg\vcpkg install sfml:x64-windows
```

Потім у папці проекту:

```powershell
cmake -S . -B build -DCMAKE_TOOLCHAIN_FILE=C:\vcpkg\scripts\buildsystems\vcpkg.cmake
cmake --build build --config Release
cd build\Release
.\main.exe
```

Якщо SFML встановлений вручну з архіву, треба вказати CMake, де лежить `SFMLConfig.cmake`:

```powershell
cmake -S . -B build -DSFML_DIR="C:\SFML\lib\cmake\SFML"
cmake --build build --config Release
cd build\Release
.\main.exe
```

На Windows важливо, щоб SFML збігався з компілятором. Якщо збираєш Visual Studio 2022, бери SFML під Visual C++ 17 / 2022. Якщо MinGW - тоді SFML має бути зібраний під той самий MinGW.

## Якщо CMake не знаходить SFML

Типова помилка виглядає приблизно так:

```text
Could not find a configuration file for package "SFML"
```

Що перевірити:

| Симптом | Що робити |
| --- | --- |
| SFML не встановлений | Встановити SFML 3 |
| Встановлений SFML 2.x | Поставити SFML 3.x |
| SFML є, але CMake його не бачить | Передати `-DSFML_DIR=.../lib/cmake/SFML` |
| На Windows програма збирається, але не стартує | Перевірити, що SFML `.dll` доступні поруч з `.exe` або в `PATH` |
| Shader не завантажується | Запускати програму з build-папки, де лежить `mandelbrot.frag` |

## Коротко про збірку

`CMakeLists.txt` робить три речі:

1. Шукає SFML 3:

```cmake
find_package(SFML 3 COMPONENTS Graphics Window System REQUIRED)
```

2. Збирає executable з `src/main.cpp`:

```cmake
add_executable(main src/main.cpp)
```

3. Після збірки копіює assets поруч з програмою:

```cmake
assets/mandelbrot.frag -> build/mandelbrot.frag
assets/font.ttf        -> build/font.ttf
```

Через це команда `./main` з папки `build` одразу знаходить і shader, і шрифт.
